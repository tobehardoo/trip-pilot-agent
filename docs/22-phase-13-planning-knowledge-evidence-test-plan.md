# Phase 13 规划知识证据测试计划

## 1. 目标与范围

本阶段把已审核发布的知识真正接入规划结果：先用固定广州评测集选择真实 Embedding/Rerank 配置，再由 Python Worker 生成带引用与 freshness 的 v4 完成事件，Java 按行程版本不可变保存实际使用的文档/片段版本，Vue 展示推荐依据与来源状态。

本阶段不允许把 `demo-hash-v1` 的离线结果作为语义质量结论；外部模型或数据库不可用时必须显式标记 `DEMO`/`UNAVAILABLE`，不能伪造引用。

## 2. 第一小步契约

- 广州检索评测集使用版本化 TOML，固定保存 case ID、自然语言查询、城市、可选分类和期望文档 ID；重复 case、空期望集合和非法字段必须在运行前拒绝。
- `KnowledgeRetrievalEvaluator` 只依赖 Embedding 与 Search Protocol，按 case 执行相同 `top_k` 检索，输出 Recall@K、MRR、命中数、模型名、维度和逐 case 排名；排序与浮点汇总必须确定性。
- 模型名以 `demo-` 开头时报告固定为 `DEMO_ONLY`，即使指标达到阈值也不能通过真实质量门禁；真实模型只有同时达到配置的 Recall@K 与 MRR 阈值才返回 `PASSED`。
- 评测 CLI 输出结构化 JSON，数据库或模型失败返回脱敏错误与非零退出码。真实供应商的模型、维度、批量、延迟和费用必须在同一评测集上记录后才能成为生产默认值。

## 3. 后续纵向边界

- v4 `PLANNING_COMPLETED` 必须携带知识状态、查询、引用和 freshness；引用至少保留文档 ID/版本、片段 ID/序号、标题、来源 URL/名称、采集时间、可信等级与相似度。
- Java 必须在同一完成事务中保存行程版本与引用快照，重复事件幂等，API 只返回该版本实际使用的引用。
- Vue 必须区分真实来源、Demo 与不可用状态，展示可点击来源和采集时间；动态事实仍要求实时 Provider 或明确拒答。

## 4. 第一小步结果

新增 6 项评测/知识仓储测试并扩展 CLI 测试，覆盖版本化广州数据集、语料城市/分类一致性、严格 TOML、空期望与重复 case、固定 `as_of`/文档版本、文档级 Top-K、Recall@K、MRR、阈值失败、`DEMO_ONLY`、模型/数据库错误脱敏和质量退出码。真实 pgvector PostgreSQL 下 Python 全量 303 项通过，知识检索与采集总覆盖率 92.65%。当前测试库未导入固定语料，实际 CLI 基线为 6 case、0 命中、`DEMO_ONLY`；没有伪报真实模型质量。

## 5. 纵向切片完成记录

- Worker 已发布 `PLANNING_COMPLETED` v4：真实检索只输出文档/片段版本快照，Demo 向量不会伪造真实引用，模型或数据库故障会降级为脱敏的 `UNAVAILABLE`。
- freshness 只聚合本次实际引用的已注册资源 URL；未知来源明确为 `UNAVAILABLE`，不会借用同城其他来源的状态。
- Java 已保持 v1-v3 向后兼容，并在完成事务内将行程版本、知识状态、freshness 与有序引用快照一起持久化；重复事件保持幂等。
- Vue 已展示“推荐依据”、可点击来源、来源版本、采集时间以及真实、过期、Demo 和不可用状态。
- 2026-07-23 门禁结果：Python 317 项全绿、知识检索与采集行覆盖率 92.27%；Java 91 项全绿并通过 JaCoCo 80% 门禁；Vue 47 项全绿、总行覆盖率 92.66%，生产构建成功。
- 尚未满足的外部条件：需要生产供应商凭据和已导入的固定广州语料，才能用相同评测集决定真实 Embedding/Rerank 生产默认值；在此之前系统明确报告 Demo 或不可用状态。
