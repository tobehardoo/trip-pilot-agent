# Phase 4 测试计划：完成事件、行程版本与 SSE

## 范围

- Java 消费 `PLANNING_COMPLETED` v1 事件。
- 在同一事务内完成消息去重、任务状态更新、任务事件和不可变行程版本持久化。
- 最终行程按版本、日期和活动保存为关系数据，并提供当前行程读取 API。
- `GET /api/planning-tasks/{taskId}/events` 提供所有者隔离的 SSE 历史补发与实时通知。
- 任务创建时持久化 `PLANNING_QUEUED`，完成时持久化 `PLANNING_COMPLETED`。

本阶段不包含 Python 进度/失败事件、前端时间轴渲染、版本差异与回滚、取消任务，以及 Redis 多实例 SSE 广播。

## 业务规则

- RabbitMQ 是至少一次投递；重复 `eventId` 不得重复创建任务事件或行程版本。
- 每个规划任务最多生成一个行程版本，数据库唯一约束是最终防线。
- 完成事件的 `taskId`、`tripId` 和 `traceId` 必须与任务一致。
- 任务基线旅行版本已过期时，结果标记为失败，不创建或激活行程版本。
- 行程版本、日期、活动、任务状态和任务事件必须原子提交。
- SSE 只能由任务所属用户订阅；`Last-Event-ID` 之后的事件按递增 ID 补发。

## 必须覆盖

- 首次完成创建一个 `itinerary`、一个版本、对应日期与活动，并把任务更新为 `SUCCEEDED`。
- 相同完成事件重复处理后，各表行数和当前版本保持不变。
- 行程版本保存规划时的约束快照，不能读取完成时已变化的约束作为快照。
- 关系数据写入失败时，任务状态、版本和完成事件全部回滚。
- 非法 JSON、未知字段、非法时间范围和任务标识不一致被拒绝且不重试。
- 过期基线完成事件生成一个失败任务事件，并释放活动任务唯一约束。
- 当前行程 API 隔离不同用户；没有行程时返回 404。
- SSE 补发、实时完成通知、终态关闭、跨用户 404 均通过测试。
- RabbitMQ 完成队列使用持久消息和既有死信拓扑。

## 质量门槛

- 先看到新增测试因功能缺失而失败，再编写生产实现。
- Java 自动化测试全部通过，JaCoCo 行覆盖率不低于 80%。
- Docker Compose 配置、Gitleaks、现有 Python 与 Vue 回归保持通过。
- 完成后更新 README、路线图和本文件执行结果，并推送 GitHub 等待 CI。

## 执行结果

截至 2026-07-15，本阶段后端切片已完成：

- Flyway V5 创建任务事件与不可变行程版本关系表，并验证从 V4 升级时回填约束快照和排队事件。
- Java 完成消费者严格校验消息类型、未知字段、日期/时间、标题和 `NUMERIC(12,2)` 金额边界；非法和业务冲突消息进入死信，基础设施异常保留重试语义。
- 完成写入在同一事务内锁定任务与旅行，原子保存版本、日期、活动、当前版本指针、任务终态与任务事件；重复事件和跨任务事件 ID 冲突均有回归测试。
- 当前行程 API、SSE 历史补发、实时完成、终态关闭、跨用户隔离和 `Last-Event-ID` 均通过 PostgreSQL 集成测试。
- 独立代码审查未发现 Critical 或 Important 阻塞；慢客户端发送已移出共享锁，回放与实时订阅使用任务分片锁完成无丢失交接。
