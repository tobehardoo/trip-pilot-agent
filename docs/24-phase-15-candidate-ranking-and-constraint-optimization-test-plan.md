# Phase 15 候选评分与约束优化测试计划

## 目标

把高德 POI 的“按返回顺序取前 N 个”替换为可解释候选排序，并用确定性约束求解器安排每日活动。硬约束不可行时不得降级为看似成功的 Demo 行程。

## 验收规则

- 候选按 Provider ID 以及“规范化名称 + 近似坐标”去重；空地址、城市不匹配和截断候选均保留稳定淘汰原因。
- 用户偏好命中名称、类型或地址时产生显式加分理由；同分结果按稳定键排序。
- 每日活动、步行时间和用户固定安排使用同一时间轴，固定安排作为不可移动的硬约束。
- 求解结果必须位于 09:00 至 18:00，并为相邻活动预留真实或明确标记为估算的路线时长。
- 无可行解时返回结构化冲突和最小放宽建议，不允许 `FallbackPlanningProvider` 静默生成 Demo 行程。
- 求解器限制运行时间并使用单工作线程，保证相同输入得到相同结果。

## 测试层次

- 单元测试：候选去重、城市过滤、偏好评分、稳定排序和淘汰原因。
- 单元测试：无固定安排、午间固定安排、重叠固定安排、容量不足和路线过长。
- Worker 测试：真实 Provider 使用排序后的 POI，活动避开固定安排；不可行错误穿透 Demo fallback。
- 跨服务契约测试：不可行事件可被 Java 幂等持久化并通过 SSE 给出可操作提示。

## 完成记录

候选排序与有界单线程 CP-SAT 求解器已接入真实高德规划路径；固定安排重叠、容量不足和路线占时均有确定性测试。不可行结果通过 `planning.failed` 发布，Java 幂等落库，Vue 同时展示冲突事实和放宽建议。任务取消接口会立即释放活动任务唯一约束，并拒绝晚到的完成结果。
