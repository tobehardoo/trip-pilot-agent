# 测试与 Agent 评测

## 1. 原则

- 测试投入约占总开发时间的 20%。
- 优先覆盖领域规则、消息一致性、Agent 不变量和核心用户流程。
- 不以单一代码覆盖率数字替代风险分析。
- 外部 API 测试默认使用 Demo Provider 或可控 Stub，避免消耗额度和产生不稳定结果。

## 2. Java 单元测试

- 规划任务状态流转。
- 硬约束和软约束校验。
- 行程版本创建、差异和回滚。
- 幂等键和重复消息处理。
- Outbox 状态变化。
- 权限和分享链接。
- 局部重规划影响范围。
- 费用和配额计算。

## 3. Java 集成测试

使用 Testcontainers 启动真实依赖：

- PostgreSQL/PostGIS/pgvector。
- Redis。
- RabbitMQ。

高价值场景：

> Java 事务创建任务和 Outbox，消息成功投递，模拟 Agent 返回完成事件；Java 重复消费两次相同事件，但只生成一个行程版本。

其他场景：

- Outbox 发送失败后恢复。
- 乐观锁冲突返回 409。
- Refresh Token 轮换和撤销。
- SSE 事件补发。
- 死信任务受权限控制地重放。

## 4. Python 测试

- LLM 输出 Schema 校验。
- POI 去重、过滤和评分。
- 路线矩阵构造与缓存 Key。
- OR-Tools 约束和停止条件。
- 无解时的放宽顺序。
- 局部重规划保护锁定活动。
- Provider 超时、限流、非法响应和无结果。
- Checkpoint 恢复和协作式取消。

## 5. 优化器不变量

每个返回方案必须自动检查：

```text
活动不重叠
地点不非法重复
必去地点全部存在
不超过每日时间范围
满足预约时间
交通时间占用日程
不突破硬预算
锁定活动没有移动
最后一天按时到达返程地点
每个 POI 有来源
```

不能只断言“求解器返回非空”。

## 6. 契约测试

- OpenAPI 验证 Java HTTP 接口。
- JSON Schema 验证 RabbitMQ 事件。
- Java 序列化出的事件必须能被 Python 解析。
- Python 结果必须能被 Java 解析和校验。
- 历史事件样例用于验证向后兼容。
- 时间允许时为消息补充 AsyncAPI 文档。

## 7. Playwright E2E

核心正常流程：

1. 创建广州旅行。
2. Agent 发现返程时间缺失。
3. 前端显示结构化追问。
4. 用户补充信息。
5. SSE 展示规划进度。
6. 地图和时间轴展示结果。
7. 用户删除并锁定活动。
8. Agent 只重规划当天。
9. 用户查看版本差异并回滚。
10. 创建只读分享链接。

异常流程：

- 高德接口超时。
- LLM 返回非法 JSON。
- RabbitMQ 重复投递。
- 任务运行中取消。
- SSE 断线重连。
- OR-Tools 无可行解。

## 8. Agent 评测层级

| 层级 | 评测内容 |
|---|---|
| 需求理解 | 日期、预算、偏好、必去地点等字段解析 |
| 工具使用 | 工具选择、参数、调用次数和错误处理 |
| 计划可行性 | 时间、路线、预算、预约和锁定活动 |
| 计划质量 | 偏好、通勤、丰富度、强度和解释质量 |

## 9. 指标

- 结构化约束字段准确率。
- 必去地点覆盖率。
- 硬约束违反次数。
- 不存在或错误 POI 数量。
- 预算偏差率。
- 平均每日通勤时间。
- 无解识别率。
- 局部重规划范围准确率。
- 工具调用成功率和缓存命中率。
- 单次任务 Token、费用和耗时。
- 人工偏好评分。

LLM-as-a-Judge 只能作为主观质量的辅助指标，不能判断确定性时间、预算和路线约束。

## 10. 初始评测集

先完成约 30 条：

- 广州、杭州、西安各 8 条正常规划，共 24 条。
- 信息缺失、硬约束冲突、局部重规划和外部工具异常合计至少 6 条。

随后扩充到 50 条，并平衡：

- 不同行程天数。
- 独自、情侣和朋友出行。
- 不同预算与节奏。
- 正常、冲突、无解和故障场景。

具体用例格式、输入、期望约束、检查器和评分示例在实现评测模块时补充。用户已经明确要求届时提供可直接使用的示例。

## 11. 硬性验收门槛

固定评测集上必须满足：

```text
虚构 POI = 0
硬约束静默违反 = 0
锁定活动意外移动 = 0
重复消息导致重复版本 = 0
```

其他质量指标先建立 baseline，再通过模型、Prompt、检索和权重实验改进，不在没有数据时承诺虚假数值。

## 12. CI 建议

GitHub Actions 分层执行：

1. 格式、静态检查和单元测试。
2. Java/Python 契约测试。
3. Testcontainers 集成测试。
4. Demo Provider 下的关键 E2E。
5. 固定小型 Agent 回归评测。
6. 构建容器镜像和基础安全扫描。

完整 30 至 50 条模型评测可以手动或定时执行，避免每次提交产生费用。
