# 产品完整度与需求基线

> 基线日期：2026-07-23
>
> 适用分支：`main`
>
> 状态：后续产品与工程需求的优先级依据

## 1. 结论

TripPilot 已经是一个可以部署和演示的单城市智能旅行规划 MVP，但尚未达到
`docs/00-project-charter.md` 定义的完整 V1。

- 相对“可演示 MVP”的完成度：约 **90%**。
- 相对“项目章程完整 V1”的用户功能完成度：约 **63%**。
- 相对“可长期维护和运营的生产工程”的工程完成度：约 **78%**。
- 按用户价值 60%、工程能力 40% 加权，项目整体完成度约 **69%**，对外可简写为
  **约 70%**。

这些数字不是代码行数或测试覆盖率，而是按已实现、部分实现、尚未实现的用户闭环和
生产能力加权得到。只有验收标准通过后才提高对应维度的分数。

## 2. 用户视角：现在能做什么

当前版本可以让用户：

1. 注册、登录、恢复会话并安全退出。
2. 创建单城市旅行，填写日期、预算、人数、同行类型、旅行节奏、偏好和固定安排。
3. 发起异步规划，在页面查看进度、取消任务，并在断线后继续接收事件。
4. 使用真实高德 POI 和步行路线生成多日行程；无 Key 时使用明确标记的 Demo 数据。
5. 让 OR-Tools 校验时间、交通、预算和固定安排；不可行时看到冲突原因和放宽建议。
6. 在时间轴和地图中查看地点、顺序、通勤路线、费用估算和数据来源。
7. 查看规划使用的城市知识来源及其新鲜度。
8. 向当前旅行导入无需登录即可访问的公开攻略，查看其中的景点、餐饮、交通、费用、
   排队和预约事实，以及原文、采集时间、置信度和有效期。
9. 获得账号级、旅行级数据隔离；其他用户不能读取当前旅行及其攻略。

一句话描述：**当前版本能够生成一份真实地点驱动、受约束校验、带来源说明的单城市
多日行程，并让用户补充和核对公开攻略情报。**

## 3. 开发视角：现在具备什么

当前工程已经具备：

- Vue 3 + TypeScript + Nginx 的旅行工作台、地图、时间轴、SSE 和攻略情报界面。
- Spring Boot + Spring Security + MyBatis + Flyway 的身份、旅行、规划、行程版本和
  攻略数据边界。
- JWT Access Token、HttpOnly Refresh Cookie 轮换、CSP、接口限流和用户所有权校验。
- Transactional Outbox、RabbitMQ、幂等消费、取消命令、死信队列和事件历史补发。
- Python 规划 Worker、Provider Protocol、Redis 缓存、候选过滤/去重/排序和 OR-Tools
  约束求解。
- 高德 POI 与步行路线 Provider，以及不会伪装成真实数据的 Demo 降级。
- PostgreSQL、pgvector、版本化知识导入、引用、新鲜度和受审核的官方资料采集链路。
- 用户提交攻略的 SSRF 防护、DNS 公网校验、同域重定向、正文/事实上限、内部服务令牌、
  边缘限流和行程级持久化。
- Java、Python、Web 自动化测试与覆盖率门禁、敏感信息扫描、生产镜像构建、
  Docker Compose、Prometheus 和数据库备份/恢复脚本。

一句话描述：**当前版本已经形成清晰的 Web—业务后端—消息队列—规划 Worker—数据层
端到端架构，核心链路具备测试、幂等、安全和降级能力。**

## 5. 用户视角：主要缺口与建议

按用户价值排序：

1. **攻略没有真正参与规划。** 当前攻略事实只在旅行详情中展示，不会影响地点候选、
   排序或约束。建议允许用户选择启用的攻略，并在新行程中看到“哪条事实影响了哪个决定”。
2. **旅行上下文还不够完整。** 缺少到达/返程地点和时间、酒店或住宿区域、必去地点、
   排除地点、用餐时间、行动能力等关键输入。
3. **生成后不能直接修改。** 用户需要删除、移动和锁定活动，并只重新规划受影响时段。
4. **版本能力没有产品化。** 用户看不到两个版本的差异，也不能安全回滚。
5. **动态信息不足。** 天气、营业时间、预约要求、票价和公共交通应在生成时与出发前复核。
6. **攻略生命周期不完整。** 缺少删除导入、手动重新核验、失效来源提示和来源启用/停用。
7. **结果不便分享和携带。** 缺少只读分享、PDF/日历导出和离线查看。
8. **缺少多方案比较。** 当前主要返回一个方案；用户需要在轻松、均衡、密集或不同预算方案间比较。
9. **移动端仍以查看为主。** 需要更适合旅途中的导航、今日行程和弱网使用体验。

## 6. 开发视角：主要缺口与建议

1. **规划证据边界需要升级。** Java 应在任务创建时生成行程级不可变证据快照，通过版本化
   消息契约交给 Python；Python 不应直接读取 Java 业务表。
2. **需要规划命令 V2。** 增加到返、住宿锚点、必去/排除、锁定活动、变更范围和已选择攻略事实。
3. **需要局部重规划用例。** 新任务类型必须携带基线旅行版本和行程版本，保持未受影响日期不变。
4. **需要动态事实端口。** 天气、营业、票价、预约和公共交通应实现统一的超时、重试、缓存、
   TTL、来源和错误分类契约。
5. **需要跨服务 E2E 门禁。** CI 或 nightly 应覆盖浏览器—Java—RabbitMQ—Python—
   PostgreSQL—SSE 的真实链路。
6. **业务可观测性不足。** 需要任务成功率、队列积压、Provider 延迟/错误、缓存命中、
   规划耗时、约束失败、攻略新鲜度和跨服务 Trace。
7. **缺少运维操作面。** 死信查看/重放、失败任务诊断、来源健康度和配额状态目前没有受保护界面。
8. **跨语言契约仍靠手工同步。** 应从 JSON Schema 生成或至少自动校验 Java/Python 类型。
9. **前端和数据适配器需要按行为拆分。** `TripDetail.vue`、`TripDashboard.vue`、
   `map.py` 和检索 Repository 已超过舒适职责范围；必须先用行为测试锁定再重构。
10. **公开生产环境仍需部署证据。** 需要真实 HTTPS、告警、恢复演练记录和不可变镜像发布记录，
    不能只以本机 Compose 健康作为正式上线完成标准。

## 8. 实施顺序

### V1.2：让规划可编辑、可利用用户情报

执行 `TP-REQ-001` 至 `TP-REQ-004`。

关键架构规则：

- Java 继续拥有用户、旅行、攻略、任务和行程版本的业务事实。
- Java 创建规划任务时生成不可变 `planning evidence snapshot`；Python 只消费契约。
- 社区攻略事实默认只能影响候选排序和软建议，不能直接成为硬约束。
- 只有经过授权 Provider 核验的营业/闭馆/预约事实才能阻止具体安排。
- 局部重规划使用新的任务类型，不在现有 CREATE 分支中堆叠条件判断。

### V1.3：让结果可信、可恢复、可分享

执行 `TP-REQ-005` 至 `TP-REQ-010`。

重点先完成动态事实与版本回滚，再开放匿名分享；分享内容必须是不可变版本快照，不能把
当前可变旅行对象直接暴露给匿名访问。

### V1.4：让系统可长期运营

执行 `TP-REQ-011` 至 `TP-REQ-015`。

只有跨服务 E2E、业务指标、告警、失败重放和契约门禁形成闭环后，才把项目标记为
“生产可运营 V1”。

### 后续扩展

`TP-REQ-016` 至 `TP-REQ-018` 不进入当前 V1 完成条件，避免多城市、协作和库存接口扩大
尚未稳定的核心模型。

## 9. 每项需求的完成定义

所有需求必须同时满足：

1. 有用户故事、异常路径和明确的不做范围。
2. 先补失败测试，再实现最小端到端切片。
3. 用户所有权、幂等、版本冲突和数据过期行为有自动化测试。
4. Java/Python/TypeScript/JSON Schema 契约保持兼容。
5. 真实 Provider 有超时、重试、限流、缓存、错误分类和降级说明。
6. UI 展示真实/演示、实时/估算、有效/过期状态，不隐藏不确定性。
7. 文档、迁移、监控、发布和回滚步骤同步更新。
8. 独立代码审查无 Critical/Important 问题。

## 10. 完整 V1 退出条件

完成以下条件后，用户功能完整度可提升至 85% 以上，并将版本标记为完整 V1：

- 攻略和已核验动态事实能进入规划并在结果中追溯。
- 到返、住宿、必去/排除、用餐和行动能力可表达并被约束验证。
- 用户可以删除、移动、锁定活动并执行局部重规划。
- 用户可以比较和回滚行程版本。
- 天气、营业、预约、票价和公共交通有实时核验或明确拒答策略。
- 只读分享和至少一种可携带导出形式可用。
- 跨服务 E2E、业务指标、告警和恢复演练形成发布证据。
