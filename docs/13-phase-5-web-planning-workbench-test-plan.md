# Phase 5 测试计划：网页规划闭环与 Demo 行程时间轴

## 范围

- Vue 旅行工作台创建规划任务，并为每次用户操作生成新的 UUID 幂等键。
- 浏览器使用带 Bearer Token 的流式 `fetch` 订阅任务 SSE，解析历史补发与实时事件。
- 工作台展示排队、完成和失败状态；任务运行时禁止重复提交。
- 页面进入旅行详情时读取当前行程；没有当前行程时展示明确空状态。
- 完成事件到达后重新读取当前行程，并按日期、时间和活动渲染 Demo 时间轴。

本阶段不包含地图、任务取消、进度事件、版本历史、差异与回滚、拖拽编辑，以及刷新页面后恢复未完成任务。

## 业务规则

- 规划任务请求必须携带 `Idempotency-Key` 和当前 Access Token。
- SSE 请求必须携带 `Authorization: Bearer ...` 与 `Accept: text/event-stream`；浏览器原生 `EventSource` 无法满足当前 Bearer Token 鉴权约束，因此使用流式 `fetch`。
- SSE 解析必须支持跨网络分块的 `event`、`id` 与多行 `data` 字段，并忽略注释心跳。
- 收到 `PLANNING_COMPLETED` 后才能读取当前行程；收到 `PLANNING_FAILED` 或流异常时保留页面并给出可重试错误。
- 切换旅行、返回列表或退出登录后，旧任务流不得更新当前页面。
- 当前行程 404 是正常空状态；其他失败必须展示错误且不能伪装成空行程。
- 时间轴以 API 返回的日期和活动顺序为准，明确标注 Demo Provider 与估算费用。

## 必须覆盖

- API 客户端正确创建规划任务，并发送 UUID 幂等键和 Bearer Token。
- SSE 数据被拆分到多个字节块时仍只产生一次完整事件。
- `Last-Event-ID` 在重连请求中正确发送。
- 页面首次进入详情时，当前行程 404 会展示“尚未生成行程”和开始规划入口。
- 点击开始规划后按钮进入禁用状态，并显示排队状态。
- 完成事件触发当前行程读取，随后展示行程标题、版本、总费用、日期和活动时间轴。
- 失败事件与网络中断显示可操作的重试信息。
- 规划接口或 SSE 首次返回 401 时沿用现有 Refresh Token 轮换并重试一次。
- 返回列表或退出登录会中止任务流，迟到响应不污染其他旅行或会话。
- 既有认证、旅行创建、详情和约束编辑回归保持通过。

## 质量门槛

- 先看到新增测试因功能缺失而失败，再编写生产实现。
- Vue 自动化测试、TypeScript 类型检查与 Vite 生产构建全部通过。
- 使用真实浏览器检查至少一个桌面和一个移动端视口，确认时间轴、状态与操作按钮不重叠。
- 在本地真实服务可用时，完成网页创建任务到 Demo 时间轴显示的跨服务冒烟；若依赖环境不可用，明确记录未验证项。
- 完成后更新 README、路线图和本文件执行结果。

## 执行结果

截至 2026-07-16，本阶段已完成：

- 前端 API 层新增规划任务、当前行程类型与请求函数，并以带 Bearer Token 的流式 `fetch` 解析 SSE；解析覆盖跨字节分块、多行字段和 `Last-Event-ID`。
- App 统一管理任务状态、当前行程、请求序列与 `AbortController`；切换路由、会话变化或组件卸载后，旧流不能写回当前页面。
- 网络型读流错误最多执行 3 次连接尝试（初始连接和最多 2 次重连），并从最后一个已处理事件继续补发；业务 HTTP 错误不会被无条件重试。
- 工作台展示空行程、排队、失败、加载和完成状态；任务运行时禁止重复提交，完成后自动刷新当前行程。
- Demo 行程按日期和活动渲染响应式时间轴，显示 Provider、版本、估算费用和来源；API 返回的 UTC 时间按 `Asia/Shanghai` 正确显示。

最终验证结果：

- Vue：33/33 自动化测试通过，覆盖规划 POST/SSE 的 401 轮换、固定幂等键、业务失败、重试耗尽、路由中止和迟到事件隔离；TypeScript 类型检查与 Vite 生产构建通过。
- 真实跨服务浏览器冒烟：注册 -> 创建广州 4 日游 -> 创建任务 -> Outbox/RabbitMQ -> Python Demo Worker -> Java 完成消费 -> PostgreSQL 行程版本 -> SSE -> 网页时间轴，全链路通过。
- Playwright CLI：1440 x 1000 桌面与 390 x 844 移动视口截图检查通过；移动端文档宽度未超过可视区，未发现文字或控件重叠。
- 浏览器验收发现并修复 UTC 时间被直接截取导致 `01:00` 错显的问题，回归测试使用真实 `Z` 时区响应并验证显示为 `09:00`。

保留到后续阶段的非阻塞项：刷新页面后恢复仍在运行的任务，需要后端提供按旅行查询活动任务的读取接口；当前 404 空行程请求会在浏览器控制台产生预期的资源错误记录。
