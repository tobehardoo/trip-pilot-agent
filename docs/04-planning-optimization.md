# 行程优化与约束求解

## 1. 问题定义

多日旅行规划接近带时间窗、可选节点和收益的多日路径规划问题，即 Prize-Collecting VRP with Time Windows。

- 每一天可视为一辆虚拟车辆。
- 景点和活动是候选节点。
- 酒店通常是每天起点和终点。
- 景点有时间窗、服务时长、费用、偏好收益和可访问日期。
- 必去节点不可跳过，可选节点可以通过惩罚机制跳过。
- 目标是在可行前提下提高偏好收益、减少通勤和降低软约束违反。

特殊起终点：

```text
第一天：到达车站 -> 活动 -> 酒店
中间天：酒店 -> 活动 -> 酒店
最后天：酒店 -> 活动 -> 返程车站
```

## 2. 优化器输入

```text
TripOptimizationRequest
├── days
│   ├── date
│   ├── startLocation / endLocation
│   ├── availableStartTime / availableEndTime
│   └── weatherContext
├── candidatePlaces
│   ├── placeId / coordinates
│   ├── preferenceScore
│   ├── duration / durationConfidence
│   ├── openingWindows
│   ├── estimatedCost
│   ├── required
│   ├── allowedDays
│   └── indoorOutdoor
├── travelMatrix
├── budget
├── pace
├── maxWalkingTime
└── lockedActivities
```

## 3. 硬约束

- 必去地点必须安排。
- 一个地点最多访问一次。
- 活动必须落在开放或预约时间窗内。
- 活动时长和通勤时长必须占用真实日程。
- 每天不得早于开始时间或晚于结束时间。
- 第一日不得早于到达时间。
- 最后一日必须预留到车站和进站时间。
- 锁定活动不得移动或删除。
- 明确的预算硬上限不得突破。
- 明确禁止的天气和活动组合不得安排。

## 4. 软约束

- 减少总通勤时间。
- 减少步行时长。
- 每日行程强度尽量均衡。
- 避免连续过多同类型景点。
- 午餐、晚餐时间合理。
- 优先匹配用户兴趣和同行类型。
- 期望预算附近优于明显低于或高于期望。
- 下雨时优先室内活动。
- 酒店与每天首尾活动距离合理。

## 5. 目标函数

OR-Tools 最小化整数成本：

```text
总成本 =
    通勤时间成本
  + 跳过高偏好 POI 的惩罚
  + 每日强度不均衡惩罚
  + 预算偏差惩罚
  + 等待时间惩罚
  + 其他软约束惩罚
```

- 分数和成本先归一化到稳定区间，再转换为整数。
- 权重不得凭感觉写死；先设置基线，再通过评测集调整。
- 对外说明为“限定时间内寻找高质量可行解”，不声称必然得到全局最优解。

## 6. 三段式优化

```text
求解前
规则过滤、去重、置信度检查、候选评分、地理粗分组
  -> 求解中
时间窗、路线、必去节点、预算、访问选择、起终点
  -> 求解后
餐饮合理性、类别多样性、强度、业务规则和来源校验
```

不把所有语义规则强行写进数学模型。例如“连续三个博物馆体验单一”更适合候选排序和求解后校验。

## 7. 路线矩阵控制

不能对大量 POI 直接做完整两两高德查询。

推荐流程：

```text
30 至 50 个原始 POI
  -> 规则与偏好筛选至 15 至 20 个
  -> PostGIS/Haversine 粗距离和地理分组
  -> 只查询同一天候选点的真实路线
  -> OR-Tools 优化顺序
  -> 对最终相邻节点做路线精化和复检
```

Redis 路线缓存 Key 至少包含：

```text
起点坐标 + 终点坐标 + 出行方式 + 时间段 + Provider + 数据版本
```

公共交通与时间段相关，不允许永久缓存。

## 8. 数据可信度

开放时间和建议游玩时长可能不完整，采用以下优先级：

```text
人工维护的增强城市资料
  > 地图 Provider 的明确字段
  > POI 类别规则估算
  > LLM 推测
```

- LLM 推测不能成为硬约束。
- 估算字段必须带 `estimated=true` 或置信度。
- 重点 POI 可以人工维护开放时间、预约提示和建议时长。

## 9. 用餐与预约

- 午餐和晚餐作为时间窗或休息约束。
- 在确定路线后推荐附近最多 3 家餐厅。
- 默认不把某家餐厅设为不可移动的固定活动。
- 用户选择餐厅后可转为普通活动或锁定活动。
- 系统不自动预约景点。
- 用户录入的预约时间是硬约束。

## 10. 无解处理

返回无解时按固定策略分析：

1. 检查必去地点和预约时间冲突。
2. 检查返程、酒店和最后活动冲突。
3. 移除最低价值的可选活动后再次求解。
4. 尝试将个别交通段降级为出租车建议。
5. 放宽允许自动放宽的软约束。
6. 硬约束仍冲突时暂停 Agent，让用户选择。

每次放宽必须记录原因。硬约束不得静默放宽。

## 11. 求解边界

- 设置单次求解时间上限。
- 超时后可以返回当前最佳可行解并标注非最优状态。
- 无可行解时返回结构化冲突，不返回 LLM 编造的日程。
- 自动修复最多两轮。
- 局部重规划只重建受影响日期和交通矩阵。

## 12. 结果校验不变量

- 活动不重叠。
- 地点不重复，除非业务明确允许。
- 必去地点全部存在。
- 不超过每日可用时间。
- 满足预约时间。
- 包含活动间通勤时间。
- 不突破硬预算。
- 锁定活动未移动。
- 最后一天按时到达返程地点。
- POI 具有可追踪来源。

任何不变量失败都不能将版本标记为可发布行程。
